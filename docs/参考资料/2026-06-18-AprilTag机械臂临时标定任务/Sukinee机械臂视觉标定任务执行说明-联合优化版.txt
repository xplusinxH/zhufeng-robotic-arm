# Sukinee 机械臂视觉标定任务执行说明：联合优化版

## 0. 本任务的目标

本任务不是让视觉系统控制机械臂，也不是做 MoveIt 真机执行。

本任务目标是搭建一条外部视觉测量链路：

```text
固定外部相机
→ 标定后的 camera_info
→ AprilTag 检测
→ 输出末端 tag_tool0 在相机坐标系下的位姿
→ 多姿态采样
→ 和 motor_pos / q_urdf 样本对齐
→ 联合优化 motor_to_urdf software zero offset、base_link -> camera、tool0 -> tag_tool0
```

最终我们希望用视觉测量数据回答：

```text
当前 motor_pos -> q_urdf -> URDF FK 预测出的末端位置，
和真实机械臂末端位置相比误差是多少？

能否通过多姿态视觉样本修正 Joint1-Joint6 的 software zero offset？

能否同时反解：
1. base_link -> camera
2. tool0 -> tag_tool0
从而避免用手测外参？
```

本版第一阶段联合优化：

```text
1. Joint1-Joint6 的 q_urdf 零位修正量 Δq
2. base_link -> camera 的固定外参
3. tool0 -> tag_tool0 的固定外参
```

仍然不优化：

```text
1. 连杆长度
2. 关节轴方向
3. 完整 URDF 几何
4. 电机硬件零位
5. CAN_ID
6. 电机参数
7. MoveIt 真机执行
8. Gazebo effort / damping / torque / gravity_scale 真机参数
```

后续优化完成后，只输出新的 software-only offset JSON，例如：

```text
sukinee_motor_to_urdf_offsets_calibrated.json
```

不会设置电机零位，不会保存电机参数。

## 1. 坐标链路说明

### 1.1 坐标系定义

本任务使用 4 个核心坐标系：

```text
B = base_link，机器人底座坐标系
C = camera_color_optical_frame，相机光学坐标系
E = tool0，URDF 里的末端坐标系
T = tag_tool0，贴在末端刚体上的 AprilTag 坐标系
```

记号约定：

```text
T_A_B 表示：从 B 坐标系变换到 A 坐标系。
```

例如：

```text
T_C_T = camera -> tag_tool0
T_B_C = base_link -> camera
T_B_E = base_link -> tool0
T_E_T = tool0 -> tag_tool0
```

### 1.2 AprilTag 直接测出来的量

AprilTag 视觉检测直接输出的是：

```text
T_C_T_i = 第 i 个样本下 camera -> tag_tool0
```

也就是：

```text
tag_tool0 在 camera 坐标系下的位姿。
```

这个量不是手测的，也不是优化出来的，而是每个采样姿态下由视觉系统直接观测得到。

它依赖：

```text
1. 正确的 camera_info
2. 正确的 tag 实测尺寸
3. 图像中稳定检测到 tag 四个角点
4. tag 没有严重反光、模糊、弯曲或遮挡
```

### 1.3 机械臂 FK 给出的量

机械臂侧有：

```text
motor_pos -> q_urdf -> URDF FK
```

当前 software-only 映射是：

```text
q_urdf[j] = sign[j] * (motor_pos[j] - motor_pos_at_urdf_zero[j])
```

URDF / Pinocchio FK 给出：

```text
T_B_E_i = 第 i 个样本下 base_link -> tool0
```

也就是：

```text
tool0 在 base_link 坐标系下的预测位姿。
```

### 1.4 为什么不需要手测 base_link -> camera 和 tool0 -> tag_tool0

我们要联合优化两个固定外参：

```text
T_B_C = base_link -> camera
T_E_T = tool0 -> tag_tool0
```

它们不是手测值。

它们是通过多姿态数据反解出来的固定未知量。

每个样本 i 都应满足：

```text
T_B_C * T_C_T_i ≈ T_B_E_i * T_E_T
```

左边：

```text
T_B_C * T_C_T_i
=
把视觉测到的 camera -> tag_tool0 转到 base_link 坐标系下
```

右边：

```text
T_B_E_i * T_E_T
=
机械臂 FK 算出的 base_link -> tool0
再接上 tool0 -> tag_tool0 的固定安装外参
```

联合优化的目标就是让所有样本的左右两边尽量一致。

### 1.5 优化完成后如何得到视觉测量的 tool0 位姿

优化完成后，我们有：

```text
T_B_C = base_link -> camera
T_E_T = tool0 -> tag_tool0
```

相机实时测到：

```text
T_C_T = camera -> tag_tool0
```

那么视觉测量的：

```text
base_link -> tool0
```

可以由下面公式得到：

```text
T_B_E_visual = T_B_C * T_C_T * inverse(T_E_T)
```

也就是：

```text
base_link -> tool0 的视觉测量值
=
base_link -> camera
*
camera -> tag_tool0
*
tag_tool0 -> tool0
```

其中：

```text
camera -> tag_tool0：
由 AprilTag 每帧直接测量。

base_link -> camera：
由多姿态样本联合优化反解。

tag_tool0 -> tool0：
等于 inverse(tool0 -> tag_tool0)，由多姿态样本联合优化反解。
```

所以不需要 Tony 用尺子精确手测这三个变换。

Tony 和视觉同事只需要保证：

```text
1. 相机采样期间不动
2. tag_tool0 相对末端刚体不动
3. tag_base_ref 相对环境不动
4. camera_info 正确
5. tag_size_m 正确
6. 姿态样本足够丰富
7. robot sample 和 vision sample 的 sample_id 严格一致
```

## 2. 角色分工

### 2.1 Tony 负责

因为只有 Tony 有 Ubuntu 24.04 + ROS 2 Jazzy + Sukinee 机械臂环境，所以以下事情必须在 Tony 的电脑和机械臂现场完成：

```text
1. 安装和启动 ROS 2 相关软件
2. 启动 USB 相机节点
3. 加载 camera_info
4. 启动 AprilTag 检测节点
5. 启动 zero_drag 或其他安全拖动方式
6. 操作机械臂移动到采样姿态
7. 记录 Joint1-Joint6 motor_pos
8. 记录 Joint1-Joint6 q_urdf
9. 维护 sample_id
10. 保存 robot sample
11. 保存最终数据
12. 确保 MoveIt 不接真机
13. 确保视觉同事不操作 CAN 或电机
```

### 2.2 视觉同事负责

视觉同事即使没有 Ubuntu/ROS 环境，也可以负责：

```text
1. 确认 tag 打印尺寸和图案质量
2. 指导 tag_tool0 安装
3. 指导 tag_base_ref 安装
4. 指导相机安装位置
5. 检查图像质量
6. 检查 camera_info 是否合理
7. 检查 AprilTag 检测是否稳定
8. 判断每个采样姿态下 tag 是否可用
9. 检查静态 tag 抖动
10. 检查 base_ref start / middle / end 是否漂移
11. 审核 vision samples 数据质量
12. 输出视觉质量报告
```

视觉同事可以远程指导 Tony 操作 ROS 相机和 AprilTag，但不能操作 CAN、电机、MoveIt 真机执行。

## 3. 安全边界

视觉同事不要做以下事情：

```text
1. 不接 can0
2. 不控制电机
3. 不发送 Type1 torque_ff
4. 不发送 Type4 disable
5. 不发送 Type6 set zero
6. 不保存电机参数
7. 不修改 CAN_ID
8. 不启动 MoveIt 真机执行
9. 不启动 ros2_control 真机 hardware interface
10. 不让 FollowJointTrajectory 接真实机械臂
```

Tony 操作机械臂时必须保持：

```text
1. MoveIt 只允许 mock/demo
2. MoveIt 不允许真机执行
3. 不设置电机零位
4. 不保存电机参数
5. 不改 CAN_ID
6. Joint7 / left_finger / right_finger 不纳入整臂标定对象
```

stop-all 路径：

```bash
/home/zzj/sukinee_ws/src/sukinee_urdf/scripts/zero_drag/sukinee_stop_all_motors.py
```

如果使用 zero_drag 真实拖动，要明确：

```text
这涉及真实机械臂控制。
zero_drag 可能发送真实 Type1 torque_ff。
zero_drag 结束或异常时应通过 Type4 disable 路径关闭电机。
必须准备 stop-all 脚本。
MoveIt 仍不允许真机执行。
```

can0 准备命令：

```bash
cd /home/zzj/sukinee_ws

sudo ip link set can0 type can bitrate 1000000 restart-ms 100
sudo ip link set can0 up

ip -details -statistics link show can0
```

正常应看到：

```text
can state ERROR-ACTIVE
bitrate 1000000
berr-counter tx 0 rx 0
```

如果出现：

```text
Device or resource busy
```

执行：

```bash
sudo ip link set can0 down
```

然后重新执行上面的 can0 命令。

## 4. 推荐硬件与 tag 尺寸

### 4.1 相机

建议使用普通 USB 相机，优先选择：

```text
1. 能在 Linux 下稳定识别为 /dev/video*
2. 分辨率至少 1280x720，最好 1920x1080
3. 支持固定曝光更好
4. 支持固定焦距更好
5. 尽量避免自动对焦来回拉焦
6. 采样期间必须固定，不能手持
```

### 4.2 AprilTag family

建议使用：

```text
tag36h11
```

### 4.3 tag 尺寸

建议：

```text
tag_tool0: 40 mm × 40 mm
tag_base_ref: 60 mm × 60 mm
```

如果末端 40 mm 太大：

```text
tag_tool0: 30 mm × 30 mm
tag_base_ref: 50 mm × 50 mm
```

注意：

```text
这里的尺寸是 AprilTag 黑白图案主体边长，
不是整张纸边长，
不是外部白边边长。
```

打印要求：

```text
1. 100% 原尺寸打印
2. 不要“适合页面”
3. 不要自动缩放
4. 保留白边
5. 贴到硬板上
6. 保持平整
7. 不要弯曲
8. 不要反光
9. 不要皱折
10. 打印后用尺子重新测量 tag 图案主体边长
11. 记录实测尺寸，单位 m
```

tag 尺寸记录不是为了手测外参，而是为了让 AprilTag 正确估计 `camera -> tag_tool0` 的 3D pose。

## 5. tag 安装要求

### 5.1 tag_tool0 安装位置

`tag_tool0` 是末端 tag，用来测量真实末端刚体位姿。

必须满足：

```text
1. 不贴在 left_finger
2. 不贴在 right_finger
3. 不贴在任何会随 Joint7 / 夹爪开合运动的部件上
4. 固定在 Joint6 之后、但不随夹爪开合变化的刚性部件上
5. 推荐固定在 Link8 / gripper 固定座 / tool0 附近刚性安装板
6. 可以用硬纸板、泡沫板、薄亚克力、3D 打印片做小旗子支架
7. 支架必须刚性固定，不能晃动
8. 不挡夹爪开合
9. 不碰机械臂自身
10. 尽量朝向相机
```

本版不要求手工精确测量 `tool0 -> tag_tool0`，因为该外参会在联合优化中求解。

但是必须保证：

```text
tag_tool0 相对于末端刚体是固定常量。
```

如果 tag 会晃、会弯、会随 Joint7 开合，那么 `tool0 -> tag_tool0` 就不是常量，数据会失效。

### 5.2 tag_base_ref 安装位置

`tag_base_ref` 是底座附近参考 tag，用于检查相机/环境是否移动。

必须满足：

```text
1. 固定在机器人底座附近的桌面、底板或固定硬板上
2. 不贴在 Joint1 会转动的部分
3. 不要求位于 base_link 原点
4. 采样期间不能移动
5. 采样开始前、结束后、中途偶尔能看到即可
```

本版中 `tag_base_ref` 不直接用于求 `base_link -> camera`，因为 `base_link -> camera` 会由联合优化反解。

`tag_base_ref` 的主要作用是：

```text
1. 采样开始前记录 camera -> tag_base_ref
2. 采样中途记录 camera -> tag_base_ref
3. 采样结束后记录 camera -> tag_base_ref
4. 对比三者，判断相机或 base_ref 是否移动
```

### 5.3 是否要求两个 tag 同时可见

不要求。

规则是：

```text
1. 每个有效采样点必须看到 tag_tool0
2. tag_base_ref 不需要每一帧都看到
3. tag_base_ref 用于采样开始、结束、中途检查相机是否移动
4. 如果某个机械臂姿态看不到 tag_tool0，就跳过该姿态
```

## 7. 相机安装位置

```text
机械臂前上方或侧前上方
俯视角大约 30°–60°
相机固定在三脚架、支架或刚性架上
覆盖常用工作区，不追求覆盖全部极限姿态

布置要求：

4. 避免背光
5. 保持光照稳定
6. 机械臂运动后停稳再采样
7. tag 在画面中不要太小
8. tag 不要太靠近画面边缘
9. tag 不要严重倾斜到只剩很窄的透视形状
10. 相机线缆不要拉扯相机
11. 人走动或机械臂运动时不要碰到支架
```

## 11. 相机内参标定与验证

本章节不能删除，因为 AprilTag 3D pose 依赖正确 camera_info。

如果相机已经完成内参标定，本章节仍要做“加载验证”。

如果相机更换、分辨率更换、镜头焦距变化、标定 YAML 丢失，则必须重新标定。

### 11.1 为什么必须有正确 camera_info

AprilTag 即使能检测出 ID，如果没有正确 camera_info，3D pose 也不可靠。

必须保证：

```text
1. camera_info 来自当前相机
2. camera_info 对应当前分辨率
3. camera_info 对应当前焦距/镜头设置
4. tag size 使用实际测量尺寸
```

### 11.2 准备标定板

可以使用棋盘格，例如：

```text
棋盘格内角点：8x6
方格尺寸：25 mm
```

实际参数必须按真实标定板填写。

### 11.3 启动相机

```bash
source /opt/ros/jazzy/setup.bash

ros2 run v4l2_camera v4l2_camera_node \
  --ros-args \
  -p video_device:=/dev/video0 \
  -p image_size:="[1280,720]" \
  -p camera_frame_id:=camera_color_optical_frame
```

### 11.4 运行单目标定

如果图像 topic 是 `/image_raw`：

```bash
source /opt/ros/jazzy/setup.bash

ros2 run camera_calibration cameracalibrator \
  --size 8x6 \
  --square 0.025 \
  --ros-args \
  -r image:=/image_raw \
  -r camera:=/camera
```

如果图像 topic 是 `/camera/image_raw`：

```bash
ros2 run camera_calibration cameracalibrator \
  --size 8x6 \
  --square 0.025 \
  --ros-args \
  -r image:=/camera/image_raw \
  -r camera:=/camera
```

标定时移动棋盘格，让它覆盖：

```text
1. 画面左上
2. 画面右上
3. 画面左下
4. 画面右下
5. 画面中心
6. 近距离
7. 远距离
8. 不同倾斜角
```

标定完成后保存 YAML。

### 11.5 保存相机内参

建议保存到：

```bash

```

文件名建议：

```text

```

路径：

```text

```

### 11.6 启动相机时加载 camera_info

示例：

```bash

  --ros-args \
  -p video_device:=/dev/video0 \

  -p camera_info_url:=file:///home/zzj/.ros/camera_info/sukinee_usb_camera.yaml
```

检查 camera_info：

```bash

```

或：

```bash

```

验收标准：

```text
1. camera_info 里 width/height 与图像分辨率一致
2. K 矩阵不是全 0
3. D 畸变参数存在
4. P 矩阵存在
5. header.frame_id 是当前相机 frame
6. 重启相机后 camera_info 仍能加载
```

### 11.7 是否需要 image_proc

如果 AprilTag 节点使用的是 rectified image，例如 `image_rect`，需要启动 `image_proc` 生成 rectified 图像。

如果 AprilTag 节点直接使用 raw image 并配合 camera_info，也可以先按当前节点要求运行。

## 15. 静态稳定性测试

这一步必须做。否则后续采样数据不可信。

### 15.1 测试方法

让机械臂和 tag_tool0 完全静止。

条件：

```text
1. 机械臂完全静止
2. 相机完全固定
3. tag_tool0 清晰可见
4. 光照稳定
5. 不碰相机、不碰 tag
```

连续采集 100–200 帧：

```bash
ros2 topic echo /detections > /home/zzj/sukinee_ws/vision_calibration/data/static_tag_tool0_raw.txt
```

同时可以保存 TF：

```bash
ros2 run tf2_ros tf2_echo camera_color_optical_frame tag_tool0 \
  > /home/zzj/sukinee_ws/vision_calibration/data/static_tag_tool0_tf.txt
```

### 15.2 统计内容

统计 tag_tool0 pose 的：

```text
1. x 均值、标准差、最大最小
2. y 均值、标准差、最大最小
3. z 均值、标准差、最大最小
4. roll/pitch/yaw 或 quaternion 抖动
5. 检测成功率
6. 是否有明显跳变帧
```

### 15.3 验收标准

粗略标准：

```text
优秀：静止 position 抖动 1–3 mm
可用：静止 position 抖动 3–5 mm
勉强：静止 position 抖动 5–10 mm
不建议用：静止 position 抖动 > 10 mm 或经常跳变
```

如果静止 tag 的位置抖动达到厘米级，先不要进入多姿态采样，需要先排查：

```text
1. 相机内参不准
2. tag 太小
3. tag 太远
4. 图像模糊
5. 曝光不稳定
6. tag 反光
7. 相机支架晃动
8. tag 支架晃动
9. tag size 填错
10. camera_info 分辨率和实际图像分辨率不一致
```

## 17. 采样策略

由于本版要联合优化：

```text
1. Joint1-Joint6 zero offset 修正量
2. base_link -> camera
3. tool0 -> tag_tool0
```

变量比原版更多，所以样本数量必须增加。

### 17.1 小规模试采样

正式采样前先采：

```text
20 个有效样本
```

目的：

```text
1. 验证 robot sample 和 vision sample 能按 sample_id 对齐
2. 验证 AprilTag pose 数据格式可读
3. 验证 FK 数据可读
4. 验证联合优化脚本能跑通
5. 验证 frame 方向没有明显反
6. 验证结果没有明显离谱
```

小样本试采不作为最终标定结果。

### 17.2 正式采样数量

试采通过后，正式采：

```text
训练样本：100 个有效样本
验证样本：20–30 个有效样本
```

最低建议：

```text
训练样本不少于 80 个
验证样本不少于 20 个
```

不建议只采 30 个，因为联合优化变量较多，样本太少容易出现外参和 joint offset 互相混淆。

### 17.3 姿态覆盖要求

样本姿态要覆盖：

```text
1. 左侧
2. 右侧
3. 中间
4. 近处
5. 远处
6. 高位
7. 中位
8. 低位
9. Joint1 有明显变化
10. Joint2 / Joint3 有明显变化
11. Joint4 / Joint5 / Joint6 有适当变化
12. tool0 的位置要变化
13. tool0 的姿态也要变化
```

特别注意：

```text
如果 tool0 姿态几乎不变，tool0 -> tag_tool0 很难被稳定反解。
因此 Joint4 / Joint5 / Joint6 也要有适当变化。
```

不要采：

```text
1. 接近碰撞的姿态
2. 接近机械限位的姿态
3. tag 看不到的姿态
4. tag 在画面边缘的姿态
5. tag 角度太斜的姿态
6. 机械臂还在晃的姿态
7. 相机可能被碰到的姿态
8. Joint7 夹爪状态变化导致 tag 相对末端变化的姿态
```

## 18. 每个样本的采样动作

每个样本按以下流程执行。

以 `S0001` 为例：

```text
1. Tony 说：准备采 S0001
2. Tony 负责把机械臂移动到一个安全姿态
3. 停止移动
4. 等待机械臂完全静止 1–2 秒
5. 视觉同事确认 tag_tool0 清晰可见
6. 视觉侧连续取 10–20 帧 camera -> tag_tool0
7. 对 10–20 帧做中位数或稳健平均
8. 视觉侧保存 S0001 的 vision sample
9. Tony 记录 S0001 的 motor_pos 和 q_urdf
10. Tony 保存 S0001 的 robot sample
11. 双方确认 S0001 完成
12. 进入 S0002
```

sample_id 命名建议：

```text
S0001
S0002
S0003
...
```

视觉侧和机械臂侧必须使用相同 sample_id。

不要出现：

```text
视觉侧叫 S0012，robot 侧叫 S0013
```

## 22. 联合优化结果如何写回 software offset

优化脚本建议优化：

```text
Δq1, Δq2, Δq3, Δq4, Δq5, Δq6
```

含义：

```text
q_corrected[j] = q_current[j] + Δq[j]
```

当前公式：

```text
q_current[j] = sign[j] * (motor_pos[j] - zero_old[j])
```

优化后写回新的 software zero：

```text
zero_new[j] = zero_old[j] - sign[j] * Δq[j]
```

注意：

```text
这只是生成新的 software-only offset JSON。
不是设置电机零位。
不是 Type6 set zero。
不是保存电机参数。
```

## 27. 最终验收标准

```text

3. camera_info 已经在 ROS 中验证加载
4. tag_tool0 固定在末端刚体上
5. tag_tool0 不随 Joint7 夹爪运动
6. tag_base_ref 固定不动
7. /detections 稳定输出
8. /tf 能输出 camera -> tag_tool0
9. 静止 tag 抖动可接受
10. base_ref start / middle / end 没有明显漂移
11. 小样本 20 个能跑通数据链路
12. 正式训练集 80–100 个以上有效样本
13. 正式验证集 20–30 个有效样本
14. 每个有效样本都有 sample_id
15. 每个有效样本能和 robot sample 对齐
16. 数据质量足够用于联合优化
17. validation after error 明显小于 validation before error
18. 优化出的 Δq 合理
19. 优化出的 base_link -> camera 大致符合真实相机位置
20. 优化出的 tool0 -> tag_tool0 大致符合 tag 实际安装位置
```

## 28. 不可信结果的典型表现

如果优化结果出现以下情况，不要直接使用 calibrated offset：

```text
1. validation 误差没有下降
2. train 误差下降但 validation 误差不下降
3. 某个 Δq 大到离谱，例如超过 0.5 rad，需要重点排查
4. tool0 -> tag_tool0 偏移几百毫米，但 tag 实际就在末端附近
5. base_link -> camera 跑到明显不可能的位置
6. 大量样本残差异常大
7. 残差集中在某些 sample_id，怀疑 sample_id 错位
8. base_ref 显示相机采样期间移动过
9. tag_tool0 支架照片显示可能晃动
10. camera_info 分辨率和实际图像分辨率不一致
```

出现这些问题时，优先排查：

```text
1. sample_id 是否错位
2. camera_info 是否正确
3. tag_size_m 是否写错
4. tag frame 方向是否理解反了
5. quaternion 顺序是否 xyzw
6. FK 使用的 URDF 是否为当前主 URDF
7. motor_to_urdf offset 是否为当前文件
8. tag_tool0 是否松动
9. 相机是否移动
10. Joint7 是否在采样中改变并影响 tag
```
